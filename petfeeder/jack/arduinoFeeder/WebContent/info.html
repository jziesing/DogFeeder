Feeder

We built a dog feeder with communication through the internet.

Design influences

  • Arduino Yun
  • 3D printer files
  • Tomcat Server
  • UVC camera

Initial hardware build

We started with the auger based implementationon using the 3d printer to print the auger and its holder. We attatched it inside a T shaped PBC pipe to add structure and direction for the food to flow

Initial Software

The first software we worte was pretty basic. Having no experience with an Arduino Yun, but some experience with an Aduino was interesting and required a few days to adapt. We first dug into the Arduino's libraries and used a server package that handled all the requests directly with the user. This was quickly proven to be inaffective when trying to get and post mid to larger sized pieces of data. This issue led us to discover the full use of the Yun's linux processor, which has an OpenWrt image on it. The First software we ended up writing for a basic feed command used a basic javascript page that posted, a tomcat servlet, and an Arduino 'sketch' that executed get's on the server and moved the servo accordingly.

Hardware Problems

Jamming:

Caused lots of problems. Initially we tested with cerial (raisin brand) and it worked well, but when we moved to dog food and used a large container filled up with got jamming. We tried a few different things both design oriented and programming oriented, but in the end we found making the auger narrower and changing the area where food enters the auger was most effec

Camera Joystick:

This was tricky to make and could be made a lot better. Using wood for this is not ideal. I think in the future I would design some 3d files to print to make the design nicer and more effective. It does work however and proves the concept.

Audio Issue

When we set out to add audio communication to the feeder we quickly learned that a video chat like feature would be very difficult and possibly not possible. We then tried to add sending recorded messages to the feeder, and got fairly far, but failed in the end. The issue was sending the video file. We only tried on a tomcat server, which may be prohibiting us. Nevertheless I feel this feature is worth revisiting. We did think about adding some pre loaded sounds to play for your dog but found it sill when testing.

Wifi

Initially we set out to make a wifi dog feeder, since that would be more convieniet for everyone. It turned out that we ran into trouble using wifi when implementing the webcam stream. In order to do so we needed reconfigure our wifi to get our own ip. This is doable, but but for convience we decided to go with an ethernet connection, especially since things are already pretty complicated using the schools wifi.

Final Software

The final ui and front end was done using bootstrap and backbone. We settled on 4 main features: feed, schedule feed, live stream, and moving the camera. We also had a tomcat server running. This communicated with the arduino, which ended up having a bit of code on it. The Arduino turned out to be sketch code (.ino) and terminal commands, and we tried to do as much work on our server to keep better performance from the Arduino.

Video Streaming

                  Fairly easier than expected, but requires lots of gets and an idependent ip. The openwrt system has a streaming package that is great for streaming a uvc webcam, but unfortunetly not able to stream audio. We also tried to send a recorded audio clip to the arduino but ran into trouble sending the mp3 file from the server to the arduino. This may be possible but will require a lot more time to understand the Yun

Video Controls

This feature was added later when wer realized sending audio would not be possible. We knew the code for this implementation would not be hard, so we got busy with the hardware design. The design started with cardboard, then moved to wood. Although it was tricky with wood, we created a decent prototype. After attaching the camera mod we began to write code and things went pretty well. We had to account for the servo's jerky movement which was fun. This could be much better if we created the hardware with the 3d printer and wrote code to allow multiple direction movement at the same time via the keyboard.
